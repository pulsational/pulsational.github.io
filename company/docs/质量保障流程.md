## 一、交叉 Review 制度

- 每个任务必须由**非开发者**做 code review + 功能测试
- TroiBot 分配任务时同时指定：**开发者** + **Reviewer**
- 不固定搭配，按当前负载轮流分配
- 小任务至少 1 人 review，大任务建议 2 人 review

---

## 二、标准质量关卡

每个任务从开发到交付，必须经过以下流程：

```
开发完成
  → 单元测试通过
  → 提交状态改为 IN_REVIEW
  → 指定 Reviewer 做 code review + 功能测试
  → Review 通过 → TroiBot 最终验收
  → 验收通过 → 状态改为 DONE
  → 不通过任一环节 → 状态改为 NEEDS_FIX，附具体反馈，打回重做
```

### 关键规则

- **每一步必须提供实际证据**：测试输出、git diff、日志截图
- **"应该没问题"、"我觉得可以" = 不合格**，必须打回重做
- 开发者提交 review 时必须包含：文件路径、commit hash、测试结果
- Reviewer 做完 review 必须给出：通过/不通过 + 具体理由

---

## 三、任务分配规范

TroiBot 分配任务时使用以下模板：

```
任务：[具体描述]
背景：[为什么要做这个]
开发者：[指定人]
Reviewer：[指定人，不能是开发者本人]
验收标准：[怎样算完成]
使用工具：[指定 skill/agent]
涉及文件/模块：[明确范围，避免冲突]
```

### 并行任务冲突预防

- 分配并行任务时，必须确保不同开发者负责**不同文件/模块**
- 如任务有文件重叠，改为串行分配
- 需要时使用 git worktree 做隔离开发

---

## 四、上下文与协作规范

### 任务状态跟踪

- 任务完成后**必须更新** `tasks.md` 状态
- tasks.md 字段包含：ID、任务描述、开发者、Reviewer、状态、Commit、创建时间、更新时间

### 技术决策记录

- 重要技术决策**必须记录**到 `decisions.md`
- 包含：背景、决策内容、决策人、考虑过的替代方案

### 沟通规范

- 涉及公共文件改动，必须在 #company 频道 @相关人通知
- 任务完成汇报必须遵循共享上下文协议（文件路径 + commit hash + 测试结果 + git diff）

---

## 五、任务状态定义

| 状态 | 含义 |
|------|------|
| `TODO` | 待分配或未开始 |
| `IN_PROGRESS` | 开发进行中 |
| `IN_REVIEW` | 已提交，等待 Reviewer 审查 |
| `NEEDS_FIX` | Review 未通过，需修复 |
| `DONE` | TroiBot 验收通过，任务完成 |

## 六、流程图

```
Boss 下达任务
  → TroiBot 分析、拆分、分配（指定开发者 + Reviewer）
  → 开发者实现 + 单元测试 → 提交 IN_REVIEW
  → Reviewer code review + 功能测试
    → 不通过 → NEEDS_FIX → 开发者修复 → 重新提交
    → 通过 → TroiBot 最终验收
      → 不通过 → NEEDS_FIX
      → 通过 → DONE → 向 Boss 汇报
```

## 七、Scope Inventory 强制字段（v1.1 新增 · 2026-04-13 retro）

**动机**：连续 3 次同类 scope 遗漏（999/5 文案、键盘遮挡、P4-5 pause/resume）证明"只改被点名文件"是系统性漏改根因。

### 派活阶段（TroiBot）
- bug 类 / scope 类 / 同 pattern 批量修改类任务，派活前必须先跑：
  ```bash
  rg "<pattern>" App/
  ```
- 把所有命中点列入任务，每处标 **"改 / 不改 + 一句原因"**
- 不填 Scope Inventory 的 bug 任务 → 不准 IN_PROGRESS

### 开发阶段（Assignee）
- 按清单逐项勾选处理，确认每一处的决定与清单一致
- 发现新的命中点（grep 没覆盖到）→ 先回派活人补清单，再动手

### Review 阶段（Reviewer）
- 清单任何一项未打勾 / 未说明 → 直接 block，返回 NEEDS_FIX
- 新改动实际落在重复逻辑中（见 §八）→ 直接 block 要求合并源头

## 八、删除式重构优先（v1.1 新增 · 2026-04-13 retro）

**动机**：Task T 把 ConfessionView 并入 SanctuaryView 净减 1062 行、70 tests 零回归，证明合并源头比双向 patch 更根治问题。

### 规则
- 发现 2+ 处镜像同一逻辑（例如两个 View 处理同类 pause/resume 流程）→ **默认第一反应合并**，而不是分别 patch
- 只有当合并成本 >> 维护两份的成本（例如两处业务域确实不同）才保留双份，且必须在 PR 描述写明"为何不合并"
- **Reviewer 权限**：发现新改动落在重复逻辑里 → 直接 block PR，要求先合并再改，拒绝"先补 patch 下个 ticket 再合并"

## 九、证据优先报告（v1.1 新增 · 2026-04-13 retro）

**动机**：04-10 usvisa 事故后口头试行，效果好，正式成文。

### 报完成必须包含
1. **Commit hash**（完整 SHA 或至少 7 位前缀）
2. **Files changed**（文件路径清单）
3. **Test 证据**：命令 + 输出尾巴（tests passed 行、失败数）或 screenshot
4. **Scope Inventory 对照**：清单每一项实际如何处理
5. **已知残留风险**（若有）

### TroiBot 行为
- 收到报告后**独立 `git diff <hash>` 核查**，不靠口头承诺
- 缺项 → 拒收，返回 NEEDS_FIX
- 模板见 `docs/report-template.md`

## 十、自启项登记（v1.1 新增 · 2026-04-13 retro）

**动机**：ToniBot 擅装 LaunchAgent + 残留 CompanyTerminal 登录项 → Boss 重启被 CRITICAL 轰炸。

### 规则
- 任何 `LaunchAgents/*.plist`、`RunAtLoad=true`、`osascript ... login items add`、brew services、Helper app 等自动启动机制，**动手前必须**：
  1. 在 `docs/autostart-registry.md` 登记目的 + 文件路径 + 触发条件
  2. 在 #company @Boss 说明并等 approve
  3. approve 后再执行
- **无登记 = 无权执行**。Reviewer / TroiBot 发现未登记的自启项 → 立即停用 + 上报 Boss
